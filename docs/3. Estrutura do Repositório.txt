==================================================
ESTRUTURA IDEAL DO REPOSITÓRIO GIT
==================================================

Objetivo:
Organizar a plataforma de infraestrutura de forma:

- modular
- reproduzível
- segura
- escalável
- fácil de manter
- preparada para automação futura

A estrutura deve facilitar:
- versionamento
- auditoria
- backup
- troubleshooting
- CI/CD
- segregação de responsabilidades

==================================================
DESCRIÇÃO DOS DIRETÓRIOS
==================================================


--------------------------------------------------
/infra
--------------------------------------------------

Responsável por toda infraestrutura da plataforma.

Contém:
- stacks Docker Compose
- configurações de serviços
- arquivos persistentes
- exports
- componentes de autenticação
- observabilidade
- automações locais


--------------------------------------------------
/infra/cloudflared
--------------------------------------------------

Função:
- túnel seguro Cloudflare
- publicação controlada dos serviços
- integração com Cloudflare Zero Trust

Arquivos:
- docker-compose.yml
- config.yml

Responsabilidades:
- exposição segura dos serviços
- isolamento do ambiente interno
- eliminação de portas públicas desnecessárias


--------------------------------------------------
/infra/nginx
--------------------------------------------------

Função:
- reverse proxy
- TLS termination
- roteamento interno
- headers de segurança

Arquivos:
- docker-compose.yml

Subdiretórios:
- /conf.d
- /ssl

Responsabilidades:
- proxy reverso seguro
- integração com Keycloak
- segurança HTTP
- políticas de acesso
- hardening de headers


--------------------------------------------------
/infra/keycloak
--------------------------------------------------

Função:
- IAM corporativo
- SSO
- MFA
- federação de identidade

Arquivos:
- docker-compose.yml
- realm-export.json

Responsabilidades:
- autenticação centralizada
- autorização
- gerenciamento de usuários
- MFA obrigatório
- integração OAuth2/OIDC/SAML


--------------------------------------------------
/infra/postgres-auth
--------------------------------------------------

Função:
- persistência dos serviços críticos
- armazenamento do Keycloak
- armazenamento do Infisical

Arquivos:
- docker-compose.yml
- backup.sh

Responsabilidades:
- persistência segura
- backup automatizado
- restauração operacional
- retenção de dados

Importante:
- nunca expor publicamente
- acesso apenas por rede interna


--------------------------------------------------
/infra/redis-auth
--------------------------------------------------

Função:
- cache
- sessões
- armazenamento temporário

Arquivos:
- docker-compose.yml

Responsabilidades:
- performance
- gerenciamento de sessões
- aceleração do IAM

Importante:
- acesso interno apenas
- nunca expor externamente


--------------------------------------------------
/infra/smtp-relay
--------------------------------------------------

Função:
- relay SMTP OAuth2
- integração Microsoft 365
- envio seguro de e-mails

Arquivos:
- docker-compose.yml

Responsabilidades:
- envio de MFA
- notificações
- integração corporativa

Importante:
- aceitar SMTP apenas internamente
- bloquear SMTP externo


--------------------------------------------------
/infra/infisical
--------------------------------------------------

Função:
- gerenciamento centralizado de secrets
- rotação de credenciais
- distribuição segura de secrets

Arquivos:
- docker-compose.yml

Responsabilidades:
- substituir .env inseguros
- centralizar segredos
- auditoria de acesso
- integração segura com aplicações


--------------------------------------------------
/infra/monitoring
--------------------------------------------------

Função:
- observabilidade
- métricas
- logs
- uptime monitoring

Componentes:
- Grafana
- Loki
- Uptime Kuma

Responsabilidades:
- monitoramento operacional
- troubleshooting
- dashboards
- centralização de logs
- alertas

==================================================
RECOMENDAÇÕES IMPORTANTES
==================================================

Aplicar:

- segregação clara por responsabilidade
- stacks desacopladas
- versionamento completo
- documentação obrigatória
- automação operacional
- backup versionado
- restore testado periodicamente
- estrutura preparada para CI/CD
- separação entre infra e aplicação
- deny-by-default nas redes Docker
- containers sem privilégios desnecessários
- rootless containers quando possível
- MFA obrigatório
- SSH somente com chave
- hardening Linux
- observabilidade centralizada
