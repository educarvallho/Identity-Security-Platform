==================================================
SEGREGAÇÃO DE REDES DOCKER
(Quem fala com quem)
==================================================

Objetivo
--------------------------------------------------
Aplicar segmentação real de redes utilizando o
princípio de mínimo privilégio.

Ou seja:
- cada container enxerga apenas o necessário
- reduzir superfície de ataque
- impedir comunicação desnecessária
- isolar serviços críticos


==================================================
ARQUITETURA DE REDES
==================================================

[ edge-network ]
--------------------------------------------------
- cloudflared
- nginx


[ auth-network ]
--------------------------------------------------
- nginx
- keycloak
- smtp-relay
- redis-auth
- infisical


[ db-auth-network ]
--------------------------------------------------
- keycloak
- infisical
- postgres-auth


[ monitoring-network ]
--------------------------------------------------
- grafana
- loki
- uptime-kuma
- nginx


[ backup-network ]
--------------------------------------------------
- postgres-backup
- postgres-auth
- infisical
- object-storage-client


==================================================
FLUXO DE COMUNICAÇÃO
==================================================


--------------------------------------------------
SERVIÇO: cloudflared
--------------------------------------------------

PODE acessar:
- nginx

NÃO deve acessar:
- postgres
- redis
- keycloak diretamente


--------------------------------------------------
SERVIÇO: nginx
--------------------------------------------------

PODE acessar:
- keycloak
- aplicações
- monitoring

NÃO deve acessar:
- postgres


--------------------------------------------------
SERVIÇO: keycloak
--------------------------------------------------

PODE acessar:
- postgres-auth
- redis-auth
- smtp-relay
- infisical

NÃO deve acessar:
- internet irrestrita
- banco das aplicações


--------------------------------------------------
SERVIÇO: infisical
--------------------------------------------------

PODE acessar:
- postgres-auth
- keycloak
- aplicações autenticadas

NÃO deve acessar:
- edge-network


--------------------------------------------------
SERVIÇO: postgres-auth
--------------------------------------------------

PODE acessar:
- keycloak
- infisical
- backup service

NUNCA:
- exposto externamente


--------------------------------------------------
SERVIÇO: redis-auth
--------------------------------------------------

PODE acessar:
- keycloak apenas


--------------------------------------------------
SERVIÇO: smtp-relay
--------------------------------------------------

PODE:
- realizar saída HTTPS/OAuth Microsoft
- receber SMTP do Keycloak

NÃO deve:
- aceitar SMTP externo


==================================================
ACESSO À INTERNET
==================================================

Containers críticos NÃO devem possuir acesso
direto à internet.

Exemplos:
- postgres
- redis

Esses serviços não precisam de acesso externo.


--------------------------------------------------
SERVIÇOS COM SAÍDA EXTERNA PERMITIDA
--------------------------------------------------

cloudflared  -> SIM
smtp-relay   -> SIM
infisical    -> OPCIONAL
keycloak     -> OPCIONAL MÍNIMO
nginx        -> NÃO NECESSARIAMENTE


==================================================
RECOMENDAÇÃO FINAL
==================================================

Aplicar:
- isolamento por rede
- firewall interno entre containers
- deny-by-default
- acesso explícito apenas quando necessário

Objetivo:
- reduzir movimentação lateral
- limitar impacto de comprometimento
- aumentar segurança da stack
- facilitar auditoria e troubleshooting
